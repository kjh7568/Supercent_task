# Project G - Memory

## Critical Rules
- **절대 요청 범위를 초과하지 말 것**: "스탭 1만 해" → 스탭 1만 수행. 계획 전체를 한번에 진행하지 않는다.
- **커밋/MD 업데이트는 명시적 요청 시에만**: 코드 작업 완료 후 커밋·MD 수정을 임의로 하지 말 것. "ㅇㅇ 진행해"는 코드 작업만 의미.
- 단계별 작업 시, 명시적으로 요청받은 단계만 수행하고 멈출 것.
- 스탭 시작 시 반드시 아래 4가지 형식으로 설명할 것:
    - **무엇을**: 이번 스탭에서 할 작업
    - **어떻게**: 구현 방법/방식
    - **어디에**: 생성/수정할 파일 및 클래스
    - **왜**: 이 작업이 필요한 이유, 이후 어디에 쓰이는지

## Tool 사용 규칙
- **파일 텍스트 수정은 Edit 툴 사용** (Bash/sed 금지)
    - 이유: 사용자가 sed 명령을 커밋으로 오해한 사례 발생
    - Bash는 git 명령·빌드·시스템 명령에만 사용
- **모든 대화 내용을 `Assets/3.Script/all prompt.md`에 누적 저장**
    - 대화가 진행될 때마다 주고받은 내용을 append. 새 대화 시작 시에도 기존 파일에 이어서 작성.

## MD 작성 규칙
- 완료 항목 MD 표시 시 **이득을 상세하게 작성**. 수치로 표현 가능한 것은 적극적으로 수치로 표기 (예: 코드량 X줄 → Y줄, O(n) → O(1)).
- **원인/해결 섹션은 버그 수정(P1) 항목에만 작성**. 품질 개선(P2)·구조 개선(P3)에는 이득만 작성.
- 실질적 성능 체감이 없는 경우 솔직하게 MD에 명시.
- 보류 항목은 항목명 옆에 ⏸ 보류 표시 + 보류 사유 명시.
- 항목 제거·보류 시 나머지 항목 번호 갱신.

## Commit Convention
- 커밋은 명시적으로 요청받았을 때만 수행
- **단일 기능**: `기능명:S번호` (예: `선술집 용병 슬롯 표시:S2`)
- **대형 기능**: `[큰 기능명] S번호: 소제목` (예: `[임무 등급 시스템] S1: 티어 열거형 및 상수 정의`)
- **대형 기능 버그픽스**: `[큰 기능명] BF: 소제목`
- **본문**: 생성/수정한 파일명, 역할, 주요 구현 내용을 구체적으로 작성

## Player Request 기록 규칙
- 실제 구현 작업 시작 시 요청 내용 + 구현 방법을 `Player Request.md`에 저장.
- 논의·질의응답 단계에서는 저장 안 함. 코드/파일 구현 시에만 저장.
- 작업 시작 전 `Player Request.md`에서 유사 요청 확인 → 있으면 이전 방식 참고해 일관성 유지.

## 인게임 검증 규칙
- 스탭마다 인게임에서 결과를 확인할 수 있게 구현.
- 단순 데이터 구조(enum, config 등) 스탭은 예외. 기능 동작 스탭은 반드시 Debug.Log로 확인.
- **인게임 검증 없는 스탭은 다음 검증 가능한 스탭과 합칠 것.**

## 에디터 작업 안내 규칙
- 코드 작업 완료 후 Unity 인스펙터에서 설정할 내용이 있으면, 완료 메시지와 함께 **"에디터 작업"** 항목으로 정리해서 명시적으로 안내.
- 컴포넌트 연결·필드 할당·프리팹 설정 등 누락 시 런타임 null 참조 등의 문제 발생 가능.
