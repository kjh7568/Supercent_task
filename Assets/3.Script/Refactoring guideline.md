## 리팩토링 체크리스트

### 1. OOP 설계 원칙을 준수하여 설계할 것

---

### 2. 코드 세분화와 코드 통합
소스 코드를 세분화 할 수 있는 것은 세분화 하고 불필요하게 세분화 되어있는 것은 통합할 것.  
세분화는 대상(코드가 복잡하다, 재사용 가능성이 높다, 변경 가능이 높다, 의도와 구현을 분리할 필요가 있다)와 통합 대상(추상화하면 오히려 복잡해진다, 간단한 로직이 분리되어있다, 사용 지점이 한 곳 뿐이다, 함수/변수 이름이 실제 로직에 가치를 더하지 않는다)을 잘 구분해서 할 것.

* 2번의 대전제는 항상 가독성과 유지보수성을 최우선으로 고려. 코드는 컴퓨터가 실행하기 위한 것이기도 하지만, 궁극적으로는 다른 개발자들이 읽고 이해하기 위한 것

---

### 3. 객체 세분화와 객체 통합
객체 세분화(단계 쪼개기, 반복문 쪼개기)와 객체 통합(여러 함수를 클래스로 묶기, 가능하다면 여러 함수를 변환 함수로 묶기)를 통한 구현을 할 것.

결정 기준:  
세분화(책임이 명확히 분리될 때, 다른 용도로 재사용 가능할 때, 독립적으로 테스트가 필요할 때, 복잡한 단계를 나누어 명확히 할 때)  
통합(강한 응집력이 필요할 때, 관련 데이터와 동작이 함께 있는 것이 자연스러울 때, 공유 상태에 대한 관리가 필요할 때, 여러 작은 함수들이 항상 함께 사용될 때)

---

### 4. 간접접근과 직접 접근
간접 접근(변구 캡슐화, 레코드 캡슐화, 컬렉션 캡슐화)와 직접 접근(중개자 제거, 위임 숨기기)를 통한 구현을 할 것.

결정 기준:  
간접 접근(데이터 검증이나 부가 처리가 필요할 때, 변경 추적이 필요할 때, 향후 구현 변경 가능성이 있을 때, 중복된 접근 로직이 여러 개 있을 때)  
직접 접근(과도한 래퍼가 복잡성을 증가시킬 때, 성능이 중요한 핫스팟일 때, 단순한 데이터 구조일 때, 위임 체인이 너무 길어질 때)

---

### 5. 조건문과 다형성
조건문(조건식 분해 및 통합)을 사용하거나 다형성(조건부 로직을 다형성으로 바꾸기, 특이 케이스 추가하기)을 사용하는 로직을 잘 분리해서 구현할 것

결정 기준:  
조건문(단순 분기 로직, 일회성이거나 로컬 결정, 성능이 중요한 곳, 타입 배열이 고정)  
다형성(타입별 동작 차이가 뚜렷, 타입 추가가 빈번, 타입별 코드가 반복적, 동작이 확장될 가능성이 높음)

---

### 6. 상속과 위임
상속(메서드 올리기, 필드 올리기, 슈퍼 클래스 추출하기)과 위임(서브클래스 위임으로 바꾸기, 슈퍼클래스 위임으로 바꾸기)을 적절히 사용하여 의존성을 줄이는 구현을 할 것.  
단, C#에서는 다중 상속이 불가능하므로 이를 고려하여 설계할 것.

결정 기준:  
상속(명확한 Is a 관계, 공통 기능이 많고 타입 계층이 필요, 다형성을 활용한 확장이 자연스러울 때, 코드 재사용이 수직적일 때)  
위임(명확한 Has a 관계이거나 행동 공유, 런타임에서 동작이 변경되는 경우, 기존 클래스 변경 없이 기능 확장이 필요할 때)

---

### 7. 정량적 리팩토링 기준
코드 품질 판단의 일관성을 확보하기 위해 정량적인 기준을 적용할 것.  
주관적인 판단이 아닌 명확한 수치를 기반으로 리팩토링 여부를 결정할 것.

- 함수 길이가 30줄 이상일 경우 분리를 고려할 것
- 매개변수가 5개 이상일 경우 객체로 묶는 것을 고려할 것
- 조건문의 중첩 깊이가 3단계 이상일 경우 구조 개선을 고려할 것
- 동일하거나 유사한 로직이 2회 이상 반복될 경우 공통화할 것
- 클래스가 하나 이상의 책임을 수행한다고 판단될 경우 분리를 고려할 것

결정 기준:  
분리(가독성이 저하되는 경우, 함수의 책임이 명확히 구분되는 경우, 테스트 단위로 나눌 필요가 있는 경우, 변경 영향 범위를 줄일 수 있는 경우)  
통합(분리로 인해 흐름 이해가 어려워지는 경우, 단순한 로직이 과도하게 쪼개진 경우, 항상 함께 호출되는 경우, 추상화로 인해 오히려 복잡도가 증가하는 경우)

---

### 8. 의존성 관리 원칙
객체 간 의존성 방향을 명확히 하여 유지보수성과 확장성을 확보할 것.

- 상위 레이어는 하위 레이어의 구현 세부사항을 알지 않도록 설계할 것
- 순환 참조(Circular Dependency)를 절대 허용하지 않을 것
- MonoBehaviour는 순수 로직 클래스에 의존할 수 있으나, 순수 로직 클래스는 MonoBehaviour에 의존하지 않도록 설계할 것
- 의존성 주입(Dependency Injection)을 통해 결합도를 낮출 것

결정 기준:  
분리(의존성이 양방향으로 흐르는 경우, 테스트가 어려운 경우, 특정 클래스 변경 시 영향 범위가 넓은 경우)  
통합(강하게 결합된 동작이 항상 함께 실행되는 경우, 분리 시 오히려 흐름 파악이 어려워지는 경우)

---

### 9. Unity 특화 설계 규칙
Unity 환경에서의 구조적 안정성과 성능을 고려하여 설계할 것.

- MonoBehaviour는 상태 관리 및 Unity 이벤트 연결 역할만 수행하고, 핵심 로직은 일반 클래스로 분리할 것
- **Unity 라이프사이클(Awake/Start/Update 등)을 사용하지 않거나 Transform·GameObject 접근이 없다면 MonoBehaviour를 상속하지 않을 것**
- MonoBehaviour 제거 판단 기준:
  - `static` 유틸 메서드만 있는 경우 → 순수 `static class`로 전환
  - 인터페이스만 구현하고 Unity 라이프사이클을 쓰지 않는 경우 → 순수 C# 클래스로 전환
  - `Animator` 파라미터 세팅처럼 Unity 객체에 접근하되 생명주기가 필요 없는 경우 → MonoBehaviour는 얇은 래퍼로 유지하고 로직은 분리
- Update 함수 내에서 불필요한 로직 실행을 지양하고, 이벤트 기반 또는 Coroutine으로 대체할 것
- 싱글톤은 전역 상태 관리 목적에 한해서만 제한적으로 사용할 것
- UnityEngine.Object의 null 특성을 고려하여 명시적인 null 체크를 수행할 것

결정 기준:  
분리(Update에서 반복 실행되는 로직, 테스트가 필요한 로직, Unity 의존성이 없는 로직)  
통합(Unity 생명주기와 강하게 결합된 경우, 단순 연결 역할만 수행하는 경우)  
MonoBehaviour 제거(static 유틸, 인터페이스 구현체, 데이터 클래스, 순수 로직 클래스)

### 10. 데이터와 로직의 분리
데이터와 행위를 명확히 분리하여 구조의 명확성과 재사용성을 확보할 것.

- 데이터 객체(Data)는 상태 보관만 담당하고, 로직을 포함하지 않도록 할 것
- 계산 및 처리 로직은 별도의 클래스(Stat, Service 등)로 분리할 것
- Entity는 데이터와 로직을 조합하여 실제 동작을 수행하는 역할로 설계할 것

결정 기준:  
분리(데이터와 로직이 혼합되어 변경 영향 범위가 커지는 경우, 동일 데이터에 다양한 로직이 붙는 경우)  
통합(데이터와 동작이 강하게 결합되어 항상 함께 변경되는 경우)

---

### 11. 이벤트 설계 원칙
이벤트 기반 구조에서의 안정성과 예측 가능성을 확보할 것.

- 이벤트는 상태 변화의 “통지” 역할로만 사용할 것
- 이벤트 내부에서 핵심 비즈니스 로직을 수행하지 않도록 할 것
- 이벤트 구독은 반드시 해제 시점을 명확히 관리할 것
- 이벤트 흐름이 과도하게 복잡해질 경우 구조 단순화를 우선 고려할 것

결정 기준:  
분리(이벤트 체인이 길어지는 경우, 디버깅이 어려운 경우, 흐름 추적이 어려운 경우)  
통합(이벤트와 처리 로직이 항상 함께 동작하며 분리 시 오히려 이해가 어려운 경우)

---

### 12. 성능 고려 기준
리팩토링 과정에서 구조뿐만 아니라 실행 성능도 함께 고려할 것.

- GC Alloc이 발생하는 코드를 최소화할 것
- LINQ, new 연산자 사용은 핫패스에서 지양할 것
- Update 및 Coroutine의 호출 빈도와 비용을 고려할 것
- 불필요한 연산 및 반복 호출을 제거할 것

결정 기준:  
개선(프레임 드랍이 발생하는 경우, GC Alloc이 지속적으로 발생하는 경우, 호출 빈도가 높은 코드인 경우)  
유지(성능 영향이 미미하고 가독성이 더 중요한 경우)

---

### 13. 안티패턴 방지
유지보수성과 확장성을 저해하는 구조를 사전에 방지할 것.

- God Object를 생성하지 않을 것
- Feature Envy를 피할 것
- Primitive Obsession을 지양할 것
- Shotgun Surgery 구조를 방지할 것

결정 기준:  
분리(객체의 책임이 과도하게 집중된 경우, 변경 시 여러 클래스에 영향을 주는 경우)  
통합(관련 책임이 자연스럽게 하나의 객체로 묶이는 경우)

---

### 14. 네이밍 규칙
코드 가독성과 의도 전달을 명확히 하기 위해 일관된 네이밍을 사용할 것.

- 함수명은 “동사 + 목적어” 형태로 작성할 것
- 변수명은 의미를 명확히 전달하는 명사를 사용할 것
- bool 타입 변수는 is, has, can 등의 접두어를 사용할 것
- 의미 없는 축약어 사용을 지양할 것

결정 기준:  
변경(이름만 보고 역할이 이해되지 않는 경우, 오해의 소지가 있는 경우)  
유지(도메인 용어로 명확히 의미가 전달되는 경우)

---

### 15. 테스트 가능성 확보
코드의 안정성과 품질 향상을 위해 테스트 가능한 구조로 설계할 것.

- 순수 함수로 분리 가능한 로직은 최대한 분리할 것
- 외부 의존성을 최소화할 것
- 단위 테스트 작성이 가능한 구조인지 고려할 것

결정 기준:  
분리(테스트가 어려운 경우, 외부 의존성이 강한 경우, 결과 예측이 어려운 경우)  
통합(테스트 가치가 낮고 단순한 로직인 경우)
